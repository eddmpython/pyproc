# 01. 진행 원장

## 2026-07-17 - 개설과 승격

확인한 현재 상태:

1. product-gates 완결로 활성 이니셔티브 0이었다. 새 방향의 출발점은 attempts 실측이다
   (기획 전에 실측: CLAUDE.md 개발 원칙 4).
2. [branchFleet 캠페인](../../../tests/attempts/branchFleet/README.md) fleetFanOutProbe 7/7 GREEN.
   방송 팬아웃 4.05배, 4-후보 병렬 탐색 5.2배, 승계 60ms, 함대 재사용 성립.

결정:

1. 졸업 능력은 프리미티브 하나다: `forkMany(srcPid, dstPids)`. 수확 1회 + SAB 방송 +
   N 레인 병렬 적용. 이득의 근원은 "부모 델타는 값이 하나인데 N번 만들지 않는다"이다.
2. `fork(src, dst)`는 삭제하지 않는다. fork(2)는 이름 있는 개념이고 반환 계약에 게이트가
   물려 있다. `forkMany(src, [dst])`의 1:1 위임으로 남긴다(중복 구현 0).
3. Fleet 클래스(prepare/explore/adopt)는 만들지 않는다. 프리미티브 위 3줄이고 소비자가
   하나뿐이라 거짓 공통화가 된다(00-plan 참조).
4. SAB 직접 수확(메인스레드 복사 1회마저 제거)은 이번 범위 밖이다. 델타 크기를 모른 채
   버퍼를 잡아야 해서 2-pass 스캔이 필요하고, 현재 이득(4.05배) 대비 값어치가 미실측이다.
   캠페인의 다음 질문으로 남긴다.

기각:

- `fork(src, dst | dst[])` 다형 오버로드: 인자 형태로 동작 의미가 갈리는 것은 boolean
  플래그와 같은 병이다. 이름 있는 메서드 둘이 정직하다.
- 팬아웃을 map/exec에 흡수: map은 태스크 스케줄러이고 fork는 상태 복제다. 다른 계약이다.
